# 交付规范（可拼接）

## 1. 核心原则

程序侧拼接方式固定为：

```text
背景（代码生成） → 图层 A → 图层 B → … → 图层 N
```

每一张导出图必须满足：

1. **同一画布尺寸**，像素对齐，不能各张大小不一靠程序缩放凑
2. **同一套锚点**，同一风格内所有部件的「头在哪里、眼在哪里」坐标一致
3. **部件互斥**，每层只含约定内容，不把其他层的元素画进去
4. **透明底**，除该层应出现的像素外，其余区域 Alpha = 0
5. **正脸、无旋转**，头部不倾斜；否则与其他层无法对齐

画风（卡通 / 写实 / 像素 / 3D 渲染风等）由你方自定，但 **同一 `{style}` 内** 所有部件必须视觉统一，否则组合验收会失败。

---

## 2. 多风格（多套形象）怎么组织

每套形象对应一个 **style ID**（小写英文 + 连字符），例如 `devimage-cn`、`pixel-bot`。

| 项 | 规则 |
| --- | --- |
| style ID | 由产品方指定，如 `my-style` |
| 资源目录 | `packages/avatar-assets/assets/{style}/` |
| manifest | `packages/avatar-assets/manifest/{style}.json` |
| Figma | 建议一 style 一文件，或一文件多 Page 用 style 前缀区分 |

**不同 style 之间**：画风可以完全不同。  
**同一 style 内部**：必须共用 **同一份锚点模板**（见 §4），否则程序无法拼接。

---

## 3. 默认图层结构（第一期）

程序默认支持 3 个槽位（slot），从下到上：

| 顺序 | slot | 文件名目录 | 程序 z-index |
| --- | --- | --- | --- |
| 1 | `faces` | `faces/001.png …` | 最低 |
| 2 | `features` | `features/001.png …` | 中 |
| 3 | `hair` | `hair/001.png …` | 最高 |

背景圆 / 方框 / 渐变底 **不由设计师交付**，URL seed 由代码绘制。

### 3.1 各层「能画 / 不能画」

这是拼接能否成功的关键；违反任一条，组合会出现重影、露空白、层错位。

#### `faces`（脸型层）

| 必须包含 | 必须不包含 |
| --- | --- |
| 脸外轮廓、耳朵 | 头发、刘海 |
| 该 style 下的皮肤填充（颜色自定） | 眉、眼、鼻、嘴 |
| | 眼镜、帽子、项链、身体、背景 |

#### `features`（五官层）

**一张 PNG = 一套完整五官**（眉 + 双眼 + 鼻 + 嘴，画在同一张 512×512 内）。  
不是眼睛 64 张、鼻子 64 张、嘴巴 64 张分开交付。

| 必须包含 | 必须不包含 |
| --- | --- |
| 眉、眼、鼻、嘴（可缺眉） | 脸轮廓、耳朵、皮肤大色块 |
| | 头发、帽子、眼镜、背景 |

五官 **整体** 以 §4 锚点为中心绘制；各 `features/xxx.png` 之间五官位置应 **大致相同**（允许表情变化，但不能整体上下漂移）。

#### `hair`（头发层）

| 必须包含 | 必须不包含 |
| --- | --- |
| 头发、刘海、鬓角（视为发型一部分） | 脸、五官、耳朵 |
| | 帽子、头饰（除非产品单独立 slot）、背景 |

头发前缘应覆盖在发际线区域（§4），用于遮住脸层与五官层上沿的接缝。

### 3.2 降级方案（对齐困难时）

若 `features` 层无法与 `faces` 稳定对齐，可改为 **2 层结构**（需与产品确认后再改 manifest）：

| 顺序 | slot | 内容 |
| --- | --- | --- |
| 1 | `faces-full` | 脸 + 耳 + 五官一体 |
| 2 | `hair` | 仅头发 |

原则不变：**每层内容互斥、锚点一致、透明底**。

## 4. 画布与锚点（全 style 统一）

### 4.1 文件尺寸（必读）

**设计师标准交付：一张 PNG = 一个部件 = 一整张 512×512 画布。**

| 问题 | 答案 |
| --- | --- |
| 每一张导出图多大？ | **512 × 512 px**（宽 × 高） |
| 有没有「大图里切格子」？ | **没有**。不按 8×8 网格交一张大图 |
| 一行几列、每格多大？ | **不适用**——不是 Sprite Sheet，而是 **N 个独立文件** |
| 部件在文件里怎么摆？ | 头部按 §4.2 锚点落在 **512×512 全画布** 内；四周透明 |
| 能否只导出「内容最小矩形」？ | **不能**。必须导出完整 512×512 帧 |

```text
faces/001.png      → 512×512 px（仅含脸型，其余透明）
faces/002.png      → 512×512 px
features/001.png   → 512×512 px（眉+眼+鼻+嘴画在同一张内）
hair/001.png       → 512×512 px
…
```

程序叠加时，每一层 PNG **原尺寸 1:1 叠在同一 512×512 坐标系上**，不做逐张缩放或自动居中。

#### 与 Sprite Sheet（网格大图）的区别

| 方式 | 谁用 | 规格 |
| --- | --- | --- |
| **独立 PNG（本文）** | 设计师 Figma 导出 | 每张 **512×512**，共 N 张 / slot |
| Sprite Sheet | AI 批量出图 + 工程裁剪脚本 | 如 4096×4096 大图、8 行 8 列，每格裁成 512×512 |

走设计师流程时 **只交独立 PNG**；若用 AI 出网格大图，
见 [`头像美术资源规范.md`](./头像美术资源规范.md) 的裁剪流程，裁完仍是 **512×512 单张**。

### 4.2 画布属性

| 属性 | 值 | 原因 |
| --- | --- | --- |
| 尺寸 | **512 × 512 px** | 与 API `viewBox` 一致，避免运行时缩放错位 |
| 色彩空间 | sRGB | CDN 显示一致 |
| 背景 | **透明** | 程序按顺序 `composite` |

### 4.3 锚点坐标

**所有 style、所有 slot** 共用下列坐标（Figma 辅助线建议锁定，导出前对齐）：

| 锚点 | 坐标 (x, y) | 用途 |
| --- | --- | --- |
| 画布中心 | (256, 256) | 全局参考 |
| 头部中心 | (256, 260) | `faces` 轮廓中心 |
| 双眼连线中心 | (256, 220) | `features` 对齐基准 |
| 鼻尖 | (256, 250) | 验收参考 |
| 嘴巴中心 | (256, 285) | 验收参考 |
| 发际线中心 | (256, 145) | `hair` 前缘覆盖基准 |
| 头部占用区 | 椭圆 200×240，中心 (256,260) | 脸不应超出此区 |
| 安全边距 | 距四边 ≥ 32px | 防止 CDN 圆形裁切切掉内容 |

```text
y=145 ─── 发际线（hair 覆盖线）
         ╭────────╮
y=220 ── │  ●  ●  │  双眼
y=250 ── │   ▼    │  鼻
y=260 ── │  ╭──╮  │  脸椭圆中心
y=285 ── │   ∪    │  嘴
         ╰────────╯
        512 × 512
```

**对齐容差：±4 px**。超出请改源文件，程序 **不会** 逐张自动对齐。

### 4.4 视角约束

| 允许 | 不允许 |
| --- | --- |
| 正脸，双眼水平 | 侧脸、3/4 侧 |
| 头部垂直，不旋转 | 头部倾斜、俯视、仰视 |
| 对称或近对称五官 | 单眼 wink 导致左右眼 y 差 > 8px（除非全系列统一规则） |

## 5. 部件数量

数量影响「随机重复感」，与画风无关。

| 每 slot 数量 | 3 层组合总数 | 建议 |
| --- | --- | --- |
| 16 | 4,096 | 最小可上线 |
| 32 | 32,768 | 可用 |
| **64** | **262,144** | **推荐默认** |

同一 style 内，**各 slot 数量可以不同**（如 faces=32、features=64、hair=48），但需在 manifest 中声明 `count`；程序按各自数量取模。

文件编号：**三位数字，从 001 起**，与 manifest 一致。

```text
assets/{style}/faces/001.png
assets/{style}/features/064.png
assets/{style}/hair/032.png
```

## 6. 交付清单

以 style ID = `{style}` 为例：

| # | 交付物 | 说明 |
| --- | --- | --- |
| 1 | Figma 源文件 | 含锁定锚点模板 + 各 slot 部件 |
| 2 | `faces/*.png` | 数量 = 约定的 N |
| 3 | `features/*.png` | 数量 = 约定的 M |
| 4 | `hair/*.png` | 数量 = 约定的 K |
| 5 | 组合验收图 | 见 §8 |
| 6 | `style-meta.json`（可选） | 记录本 style 的 slot 数量、备注 |

**不要交付：**

- 带背景（圆/方/渐变）的 PNG
- JPG
- 未按锚点对齐的「完整头像」代替分层文件（验收图除外）
- 与约定 slot 不符的额外图层（除非产品扩展 manifest）

## 7. 导出规范

| 项 | 要求 |
| --- | --- |
| 交付形态 | **多个独立 PNG 文件**，每个文件对应一个部件编号 |
| 单文件尺寸 | **512 × 512 px**（固定，所有 slot、所有编号相同） |
| 格式 | PNG-24，带 Alpha 通道 |
| 背景 | 全透明（512×512 全区域，无内容的像素 Alpha = 0） |
| 边缘 | 无白边、无半透明白晕（抠图干净） |
| 色彩 | sRGB |
| 单张体积 | 建议 < 150KB（透明 PNG） |

**禁止**只导出内容「包围盒」尺寸（如 180×220）；**必须**导出 **512×512 完整帧**，否则与其他层无法像素对齐。

## 8. 组合验收（必须通过）

### 8.1 设计师侧（Figma）

在源文件中做 **至少 20 组** 随机组合（不同 faces + features + hair 编号），图层名记录编号，例如：

```text
QA-07 / faces:012 + features:045 + hair:003
```

### 8.2 通过标准（只检查「能否拼」，不检查画风喜好）

| # | 检查项 | 通过条件 |
| --- | --- | --- |
| 1 | 五官在脸内 | 眼、鼻、嘴落在脸轮廓内或边缘自然重合，无「浮空」 |
| 2 | 发际衔接 | 头发覆盖额头接缝，无明显断层或透明缝 |
| 3 | 耳朵 | 可被头发遮挡，但不应出现「头发切断耳朵形状」的截断感 |
| 4 | 层间无重影 | 脸层无五官线稿，五官层无脸轮廓线 |
| 5 | 随机 20 组 | **≥ 18 组** 可接受；失败组需改对应部件后重验 |

### 8.3 工程侧（可选二次验证）

资源放入仓库后：

```bash
pnpm avatar:validate -- --style {style} --samples 20
```

输出目录：`packages/avatar-assets/out/validate/`。

## 9. Figma 建议结构

与画风无关，仅便于导出与验收：

```text
📁 {style}.fig
├── Page: _Template     512 帧 + 锚点辅助线（锁定，不可删）
├── Page: faces         Component 命名 faces/001 …
├── Page: features      Component 命名 features/001 …
├── Page: hair          Component 命名 hair/001 …
└── Page: _QA           20 组叠加验收
```

每个部件 Component：

- **Frame 尺寸：512 × 512 px**（每个 Component 一帧，不是在大 Frame 里画很多小格）
- 与 `_Template` 辅助线对齐后，再转 Component
- 导出：**每个 Component 单独导出 1 张** 512×512 PNG，透明背景

```text
Figma 示例（faces slot 共 16 个部件）：
  faces/001  →  Frame 512×512  →  导出 faces/001.png (512×512)
  faces/002  →  Frame 512×512  →  导出 faces/002.png (512×512)
  …共 16 个独立 Frame，不是 1 个 2048×2048 大图里画 16 格
```

## 10. 与程序的对应关系

设计师交付 → 工程登记 manifest → API 按 seed 选各 slot 编号 → 叠图输出。

```text
seed = "张三"
  → faces:    hash(seed, "faces")    % count → 007
  → features: hash(seed, "features") % count → 023
  → hair:     hash(seed, "hair")     % count → 041
  → 按 z 顺序 composite → SVG/PNG 响应
```

因此：

- **同一 seed 必须永远得到同一组部件** → 编号稳定、文件勿随意重命名
- **部件图坐标系必须一致** → 全部 512×512 全画布导出
- **slot 内容互斥** → 程序不做「智能抠图」或「自动对齐」

manifest 示例见 `packages/avatar-assets/manifest/devimage-cn.json`。
