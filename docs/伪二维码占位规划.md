# 码形占位规划（伪 QR · 伪条形码）

> **定位**：与占位图、头像、场景图并列的第四类 **图即资源**——提供「二维码 / 一维码 / 条形码」**视觉占位**，确定性 SVG/栅格输出，**不可扫描、不编码真实业务数据**。  
> **状态**：**P1–P3 核心已落地** — `/qr`、`/barcode`、`/code/styles`；真实可扫码仍不做。

**相关文档**：[MVP 功能优先级 PRD](./MVP功能优先级PRD.md) · [完整开发文档 §5.4](./完整开发文档.md) · [竞品 URL 对照 §4](./竞品URL与API对照表.md)

## 1. 背景与场景

| 场景 | 说明 |
| ------ | ------ |
| 支付 / 登录 mock | UI 需要二维码形占位，但不应误导用户去扫 |
| 列表骨架屏 | 小程序、App 卡片中的「码」形 loading |
| 技术风文档 | 开发者文档、API 示例中的装饰性矩阵 |
| 与头像区分 | 头像圆形/角色；码形占位强调 **方形模块阵 + 定位符** 或 **条纹条形码** |
| 收银 / 物流 mock | 需要 EAN/Code128 **外形**，但不展示真实 SKU |

竞品对照：无统一国内 CDN 提供「假 QR / 假条码」占位；设计稿常用灰色方块、静态 PNG 或 Figma 组件。

## 2. 产品边界

### 2.1 做什么（占位线 · 优先）

| 类型 | 说明 | 路由草案 |
| ------ | ------ | ---------- |
| **伪 QR（二维）** | seed 确定性类 QR 视觉（定位符 + 数据区模块） | `GET /qr/:seed/:size` |
| **伪条形码（一维）** | seed 确定性条纹（Code128/EAN 外形，**非** GS1 校验） | `GET /barcode/:seed/:w/:h` |
| **样式美化** | 圆角模块、渐变底、accent 强调色、quiet zone | query `fg` / `bg` / `accent` / `radius` |
| **输出** | SVG 默认；WebP/PNG 栅格与头像一致 | 路径后缀 `.webp` / `.png` |
| **缓存** | 固定 seed URL → `Cache-Control: immutable`，CDN 友好 | 与头像 seed 路由一致 |

共性约束：尺寸 10–4000；seed 为短标识（如 `demo`、`checkout`），非业务 payload。

### 2.2 不做什么（占位线）

- **不**生成可扫描的真实 QR / 条形码（无 vCard、URL、WiFi、EAN-13 真实编码）
- **不**在 URL 中传入「待编码内容」并返回可扫图案（见 [§8 真实码生成分析](#8-真实二维码生成可行性分析暂不实现)）
- **不**对接 zxing / 扫码 SDK 做解码
- 响应头规划标注 `X-DevImage-Pseudo-Code: qr` 或 `barcode`（与真实码区分）

## 3. 路由草案

与 [竞品 URL 与 API 对照表](./竞品URL与API对照表.md) 风格一致，独立根路径（不加 `/v1`）：

| 优先级 | 路由 | 说明 |
| ------ | ------ | ------ |
| P1 | `GET /qr/:seed/:size` | 正方形伪 QR，默认 SVG |
| P1 | `GET /qr/:seed/:size.webp` / `.png` | 栅格输出 |
| P1 | `GET /barcode/:seed/:w/:h` | 伪一维码（横条为主，默认 SVG） |
| P2 | `GET /qr/:seed/:w/:h` | 矩形伪 QR（模块拉伸或留 quiet zone） |
| P2 | `?fg=&bg=&accent=&radius=` | hex 配色 + 模块圆角 |
| P2 | `?variant=matrix\|minimal\|dots` | 伪 QR 视觉变体 |
| P2 | `?variant=code128\|ean13` | 伪条码外形（仍不可扫） |
| P3 | `GET /code/styles` | 风格列表 JSON（qr + barcode 统一目录） |

**与头像实验风格关系**：

- `devimg-matrix`：头像路由内的圆形/方形裁剪实验
- `/qr/*`：独立资源，文档与 Playground 单独一页 `apps/docs/api/qr.md`

```text
apps/api/src/
├── avatar/native-renderers/matrix.renderer.ts   # 薄封装 → common/pseudo-matrix
├── common/
│   ├── pseudo-matrix.ts                         # 21×21 伪 QR 核心

│   ├── pseudo-code-colors.ts                    # fg/bg/accent query 解析
│   └── svg-raster-route.ts                      # 栅格化共享流程
└── code/                                        # 码形占位统一 Module
    ├── code.module.ts                           # /qr + /barcode + /code/styles
    ├── pseudo-code-http.ts                      # 限流常量、400 封装、StreamableFile
    ├── pseudo-code-raster.service.ts            # 伪 QR / 条码共用栅格
    ├── qr.controller.ts · qr.service.ts · qr-query.ts
    └── barcode.controller.ts · barcode.service.ts · barcode-query.ts
```

抽取建议：

- 将 `renderMatrix` 核心抽到 `common/pseudo-matrix.ts`，头像与 `/qr` 共用
- 头像版：100×100 viewBox + 可选 clip（`devimg-matrix` 保持不动）
- 占位版：任意 `size` 或 `w×h`，模块密度可随尺寸选 21 / 25 / 29
- 伪条码：用 seed 生成条宽序列 + 静区（quiet zone），**不做** Modulo-103 校验

## 8. 真实二维码生成可行性分析（暂不实现）

> **结论**：DevImage **短期不做**「URL 传入内容 → 返回可扫描 QR/条码」；与占位 CDN 定位冲突，且带来缓存、安全与合规成本。若远期有需求，应作为 **独立产品线**（如 `/encode/qr` + API Key），与伪占位路由严格分离。

### 8.1 典型 URL 方案

| 方案 | 示例 | 问题 |
| ------ | ------ | ------ |
| Query 传内容 | `GET /qr?data=https://example.com&size=256` | URL 长度受限；CDN 几乎无法复用缓存 |
| Path Base64 | `GET /qr/b64:.../256` | 仍超长；路径可读性差 |
| 短 hash + POST 创建 | `POST /encode/qr` → 返回 `id` → `GET /encode/qr/:id.png` | 需存储或纯 hash 映射；超出「零配置 img src」 |

占位 CDN 的优势是 **seed 短、URL 稳定、immutable 缓存**；真实内容在 URL 中会导致 **每个不同 payload 都是 cache miss**。

### 8.2 性能与容量

| 维度 | 伪 QR（seed） | 真实 QR（动态 data） |
| ------ | ------------- | --------------------- |
| CPU | FNV + 441 模块判定，< 1ms 级 | Reed-Solomon + 矩阵布局，通常仍 < 10ms（短文本） |
| 内存 | 纯字符串 SVG | Sharp 栅格时与头像相同，按像素线性增长 |
| CDN 命中 | 高（同 seed 永久相同） | **低**（payload 无限组合） |
| 源站 QPS | 可承受高并发 | 随机长 payload 易打满单节点 |
| URL 上限 | seed ≤ 50 字符（与 text 一致） | QR Version 40 可编码数千字符，**远超** Nginx/浏览器 header 限制 |

**注意**：即使 CPU 可接受，**缓存失效 + 带宽** 才是真实码 CDN 化的主要成本；这与 DevImage「合成图 P95 < 50ms、seed 走 CDN」的成功指标相悖。

### 8.3 安全与合规

| 风险 | 说明 |
| ------ | ------ |
| **钓鱼 QR** | 恶意链接经 `cdn.devimage.cn` 生成可扫码，用户会信任域名 |
| **支付码滥用** | 若可编码任意字符串，可能被用于代收/诱导扫码 |
| **DoS** | 超长 `data`、高 ECC、大尺寸 PNG 组合消耗 CPU/内存 |
| **内容责任** | 需可接受使用政策、举报与封禁机制（占位 seed 无此问题） |
| **误用** | 开发者把占位 CDN 当免费 QR API，与 fair-use 冲突 |

### 8.4 与伪条码 / 美化的关系

- **样式美化**（圆角、配色、logo 留白）对伪码与真实码均可复用 **SVG 后处理层**；实现占位线时预留 `style` query 即可。
- **真实一维码**（Code128 可扫）与真实 QR 面临相同缓存与滥用问题；伪条码仍应坚持 **seed 确定性、不可扫**。

### 8.5 若远期要做的前置条件（记录备查）

1. **路由隔离**：`/qr/:seed`（伪） vs `/encode/qr` 或独立子域（真），响应头区分 `X-DevImage-Encodes: 1`。
2. **长度上限**：如 `data` ≤ 500 字符；超长走 POST + 短 id。
3. **限流**：真实码单独 tier（如 Pro API Key），免费档仅伪占位。
4. **依赖**：`qrcode` / `bwip-js` 等，需评估 npm 体积与许可。
5. **合规文案**：页面与响应说明「内容由请求方提供，DevImage 不背书」。

**当前决策**：先完成 **§2 占位线**；真实码生成 **不纳入** MVP / Phase 2 范围。
